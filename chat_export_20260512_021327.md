## PRD-Ready Problem Statement

**Narrative Drift: Location Teleportation**

When Auteur drafts multiple chapters, the Cartographer outline for each chapter includes `character_state_changes` — structured records of what changed for each character (location, physical state, emotional state, etc.) per scene. After a chapter is accepted, the Bible records an event, but the current `accept` flow discards the state change data. Only `draft_iterations` is stored.

As a result, the Bible accumulates an event log that knows *that* a chapter happened, but not *where characters moved*. After 10+ chapters, a character can silently teleport from the Capital to the Dungeon with no intermediate event — and no tool catches it. The author discovers these inconsistencies during a manual re-read, often after the drift has compounded across multiple chapters.

The fix is a two-part diagnostic slice:

1. **Enrich the Bible event recording** during `accept` to persist `character_state_changes` from the Cartographer outline into event deltas and upsert character state.
2. **Build a deterministic audit rule** that reads the Bible event log, traces character locations across events, and flags consecutive events where a character's location changes with no intermediate move event.

# Test-Driven Development

## Philosophy

**Core principle**: Tests should verify behavior through public interfaces, not implementation details. Code can change entirely; tests shouldn't.

**Good tests** are integration-style: they exercise real code paths through public APIs. They describe _what_ the system does, not _how_ it does it. A good test reads like a specification - "user can checkout with valid cart" tells you exactly what capability exists. These tests survive refactors because they don't care about internal structure.

**Bad tests** are coupled to implementation. They mock internal collaborators, test private methods, or verify through external means (like querying a database directly instead of using the interface). The warning sign: your test breaks when you refactor, but behavior hasn't changed. If you rename an internal function and tests fail, those tests were testing implementation, not behavior.

See [tests.md](tests.md) for examples and [mocking.md](mocking.md) for mocking guidelines.

## Anti-Pattern: Horizontal Slices

**DO NOT write all tests first, then all implementation.** This is "horizontal slicing" - treating RED as "write all tests" and GREEN as "write all code."

This produces **crap tests**:

- Tests written in bulk test _imagined_ behavior, not _actual_ behavior
- You end up testing the _shape_ of things (data structures, function signatures) rather than user-facing behavior
- Tests become insensitive to real changes - they pass when behavior breaks, fail when behavior is fine
- You outrun your headlights, committing to test structure before understanding the implementation

**Correct approach**: Vertical slices via tracer bullets. One test → one implementation → repeat. Each test responds to what you learned from the previous cycle. Because you just wrote the code, you know exactly what behavior matters and how to verify it.

```
WRONG (horizontal):
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

RIGHT (vertical):
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## Workflow

### 1. Planning

When exploring the codebase, use the project's domain glossary so that test names and interface vocabulary match the project's language, and respect ADRs in the area you're touching.

- [ ] Design interfaces for [testability](interface-design.md)

- [ ] Get user approval on the plan

Ask: "What should the public interface look like? Which behaviors are most important to test?"

**You can't test everything.** Confirm with the user exactly which behaviors matter most. Focus testing effort on critical paths and complex logic, not every possible edge case.

### 2. Tracer Bullet

Write ONE test that confirms ONE thing about the system:

```
RED:   Write test for first behavior → test fails
GREEN: Write minimal code to pass → test passes
```

This is your tracer bullet - proves the path works end-to-end.

### 3. Incremental Loop

For each remaining behavior:

```
RED:   Write next test → fails
GREEN: Minimal code to pass → passes
```

Rules:

- One test at a time
- Only enough code to pass current test
- Don't anticipate future tests
- Keep tests focused on observable behavior

### 4. Refactor

- [ ] Extract duplication

- [ ] Run tests after each refactor step

## PRD-Ready Problem Statement

**Enhanced Init: Validate-before-init + Safe Force**

The current `auteur init` has two sharp edges:

1. **Late validation failure leaves partial state.** If the `--from` YAML is malformed or missing required fields, Pydantic raises inside `Project.init()` — which has already created the directory and `chapters/` subdirectory. The user has to manually clean up a broken project directory.

2. **No safe way to re-initialize.** `auteur init` refuses to run on an existing directory. If the user wants to reset their project (keep blueprint, wipe narrative state), they must manually delete and recreate. A single typo in the path could delete non-project files.

- Validate the blueprint YAML via Pydantic *before* touching the filesystem. Failures produce clear error messages with no artifacts.
- Add a `--force` flag that only operates on directories containing both `blueprint.yaml` and `bible.json` (auteur project detection). Non-project directories are refused.
- Print user-facing output that explains what was created and why.

### 1. CLI god module: `cli.py` at 28KB

**Files**: `src/auteur/cli.py`
**Problem**: `_cmd_audit` runs diagnostics, writes proposals, scans directories, and resolves conflicts. `_write_audit_repair_proposals`, `_load_resolved_rules`, and `_resolve_proposal` are all in `cli.py` — pure business logic with no CLI concerns, stuffed into the interface layer.

### 2. Blueprint models mixed with planning logic: `blueprint.py` at 21KB

**Files**: `src/auteur/blueprint.py`
**Problem**: This file defines 20+ enum classes, 10+ Pydantic models (the blueprint schema), AND the `PlanningCall` factory method `for_chapter()`. The `PlanningCall` is a pipeline concern — it's how the Cartographer receives blueprint projections. But it's coupled to the blueprint model via `for_chapter()` as a `@classmethod`.
**Solution**: Move `PlanningCall` and its construction logic into a separate module (e.g., `pipeline/planning_call.py` or alongside `cartographer.py`). `Blueprint` stops knowing about planning — it's just the story specification.
**Benefit**: The blueprint schema changes slower than pipeline concerns. Decoupling means changing how we project into `PlanningCall` doesn't risk breaking blueprint validation. Also makes it possible to reuse `PlanningCall` for different planners.

### 3. No outline schema validation

**Files**: `src/auteur/pipeline.py` → `_parse_outline_yaml()`, `src/auteur/cartographer.py`
**Problem**: The Cartographer prompt tells the LLM to produce a strict YAML format, but the result is parsed as `dict[str, Any]` with no Pydantic validation. A missing `scenes` key, a typo in `character_state_changes`, or a wrong type for `estimated_chapter_tension` would pass through silently — Bard would receive broken data, or a critic would crash.
**Solution**: Create a `CartographerOutline` Pydantic model matching the prompt's output schema. Validate the parsed YAML against it. If validation fails, the draft is rejected early with a clear error rather than failing downstream.
**Benefit**: Catches LLM format drift at the Cartographer boundary, not in Bard. Provides a single source of truth for the outline contract (the prompt and the schema can't disagree).

### 4. Five critics, one prompt pattern, no shared schema

**Files**: `src/auteur/critic/*.py` (7 files)
**Problem**: Each critic follows the same pattern — build a system prompt, build a user message, call LLM, parse YAML findings. `contract.py`, `arc.py`, `tension.py`, `slop.py`, `theme.py` all have near-identical structure with different prompt text. The shared infrastructure in `base.py` (`parse_findings_yaml`, `format_bible_context`) is shallow.
**Solution**: Define a `CriticPrompt` model or a `CriticRunner` class that handles the common plumbing. Each critic module becomes just a prompt provider, not a full execution flow. Two adapters for the same seam: the current pattern is one concrete adapter per prompt — that's fine, but the duplication of the run loop across 5 files is a signal the loop should be in `base.py` as a single function that receives `system_prompt` + `user_message` + `llm`.
**Benefit**: Adding a 6th critic becomes one data file instead of one module copy-pasted from the 5th. Changes to how critics call LLMs (e.g., adding retry, changing temperature, switching to structured output) happen in one place.

### 5. Proposal logic is three concerns in one file

**Files**: `src/auteur/structure/proposals.py` (24KB)
**Problem**: `proposals.py` defines the `StructureProposal` data model, the proposal *generation* logic (`propose_story_engine`, `propose_repairs_from_diagnostics`), AND the proposal *application* logic (`apply_proposal_to_blueprint`). Generation and application move in opposite directions (one generates options from a blueprint, the other merges a selected option back into a blueprint).
**Solution**: Split into `models.py` (StructureProposal + types), `generation.py` (propose_story_engine, propose_repairs), and `application.py` (apply_proposal_to_blueprint). Each gets focused tests.
**Benefit**: Generation tests don't need application fixtures. Application tests don't need generation mocks. The `--accept` resolution logic (currently in `cli.py`) can live in `application.py` where it belongs.

### 6. No LLM retry/backoff at the protocol level

**Files**: `src/auteur/llm/__init__.py` (LLMClient protocol)
**Problem**: The protocol defines `complete()` as a raw call. Transient 429s, 503s, or timeouts propagate to the CLI user as unhandled exceptions. The pipeline has no recovery path for a failed Cartographer or critic call.
**Solution**: Add a `RetryingClient` wrapper that implements `LLMClient` and wraps a delegate + retry policy (exponential backoff, jitter, max attempts). The pipeline uses `RetryingClient` by default. Provider clients stay raw.
**Benefit**: Single place to configure retry policy. Doesn't leak into provider implementations. Testable by wrapping `FakeClient` in `RetryingClient` and verifying retries on scripted failures.

### 7. `FakeClient` is list-based, not generative

**Files**: `src/auteur/llm/fake.py`
**Problem**: `FakeClient` consumes from a pre-built list of `LLMResponse` objects. For a test with 3 draft iterations (Cartographer + 6 LLM calls each), the test must pre-allocate 19 responses in the right order. Mismatches cause confusing exhaustion errors instead of clear test failures.
**Solution**: Add a `FakeClientBuilder` or make `FakeClient` accept a callable `(LLMRequest) -> LLMResponse` as an alternative to the list. Tests that care about specific call patterns can assert on the call log.
**Benefit**: Dramatically simplifies test setup for multi-step scenarios (like the 118 CLI tests which all build 7-19 response lists).

### 8. `_cmd_init` has no pre-init validation

**Problem**: As we discussed in the grill — blueprint YAML is parsed inside `Project.init()` which has already created the directory. If the YAML is malformed, the user gets partial artifacts.

## PRD-Ready Problem Statement

**Outline Schema Validation**

The Cartographer agent produces chapter outlines in a strict YAML format specified in its prompt. The format includes `scenes` (each with `scene_id`, `pov_character`, `location`, `summary`, `key_events`, `character_state_changes`, `arc_advancements`, `estimated_tension`, `emotional_tone`), plus top-level fields like `estimated_chapter_tension`, `thematic_reinforcement`, and `conflict_report`.

Currently, the outline is parsed as raw `dict[str, Any]` in `pipeline.py`'s `_parse_outline_yaml()`. No schema validation exists. If the LLM omits a required field, uses a wrong type (e.g., string instead of int for `estimated_tension`), or introduces a structural error, the pipeline doesn't catch it at the Cartographer boundary. Instead, the error manifests downstream as:

- A `KeyError` in `bard_draft()` when accessing `outline["scenes"]`
- A silent `None` in `extract_character_state_changes()` when `scenes` is missing
- A `TypeError` in `run_critics()` when a list is expected but `None` is found
- A broken `outline.yaml` written to disk that cannot be loaded by `retry`

The fix is a `CartographerOutline` Pydantic model matching the prompt's output schema. The parsed YAML is validated against it immediately after the LLM generates it. On validation failure, the pipeline rejects the draft with a clear error message instead of crashing downstream.
