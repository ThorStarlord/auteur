## Architecture

Review sessions are implemented as `src/auteur/review/` — 5 modules,
~6,000 lines. They orchestrate existing v0.8 subsystems without duplicating
them:

| Capability | Reuses from v0.8 |
|-----------|------------------|
| Decision listing | `DecisionWorkspaceService.list_decisions()` |
| Candidate detail | `DecisionWorkspaceService.inspect()` |
| Readiness/conflicts | `DecisionReadiness`, `ConflictDetector` |
| Acceptance preparation | `DecisionWorkspaceService.prepare_acceptance()` |
| Impact preview | `DecisionWorkspaceService.impact_preview()` |
| Session persistence | `ReviewStore` (append-only events under `.auteur/reviews/`) |
| CLI registration | `register_review_subcommands()` in main CLI parser |

## Session lifecycle

```
OPEN → INSPECTING → AWAITING_CHOICE → READY → PREPARED →
  AWAITING_ACCEPTANCE → ACCEPTING → ACCEPTED → REFRESHING → COMPLETED
```

Failure/recovery branches: STALE, BLOCKED, ABORTED.

## Acceptance boundary

Review session `accept()`:
- Requires `--confirm` flag (raises ValueError without it)
- Rejects stale preparation (checks `EvidenceFreshness.STALE`)
- Delegates to existing acceptance subsystem (TODO integration point marked)
- Never writes canonical pointers directly
- Only writes to `.auteur/reviews/` — no `story_identity.yaml`, `final.md`, etc.

## Event chain integrity

Events are append-only with hash chaining:

```text
event[1].hash = sha256(session_id, seq, type, prev_hash=None, payload)
event[N].hash = sha256(session_id, seq, type, prev_hash=event[N-1].hash, payload)
```

Proven:
- Tampering payload invalidates individual event hash
- Tampering hash+payload breaks link to preceding event
- Conflicting writes at same sequence number rejected
- Events survive process restart (verified via subprocess)

## Known limitations

- Acceptance execution is currently stubbed (TODO at line 426 in service.py).
  The review session architecture correctly delegates to the existing acceptance
  subsystem, but the actual call needs wiring when the full acceptance API is
  confirmed.
- `_stable_id()` includes a timestamp component (truncated to seconds), so
  logically equivalent sessions across a second boundary produce different IDs.
  This is acceptable because the `start()` method checks for active sessions
  first via the latest pointer, and the ID is only used for session identity,
  not for correctness of the resume logic.
- No graphical UI (planned for v0.10.0).
