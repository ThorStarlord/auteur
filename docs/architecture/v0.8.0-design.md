# Auteur v0.8.0 — Decision Orchestration and Evidence Integration

## Overview

v0.8.0 introduces direct subsystem integration for decisions. Instead of
indirect references through convergence proposals, the Decision Workspace
now reads directly from the reasoning, reconciliation, and impact subsystems.

## Architecture

### Subsystem Adapters

Each adapter wraps one Auteur subsystem and exposes a stable query API:

| Adapter | Source | Consumed data |
|---------|--------|--------------|
| `ReasoningAdapter` | `auteur.reasoning.draft_review` | Critic reports, synthesis reviews, source hashes |
| `ReconciliationAdapter` | `auteur.expression.reconciliation` | Proposals, conflicts, author choices |

Adapters are read-only. They are the only integration points that import
from their target subsystem directly.

### Schema Versioning

All decision artifacts carry a `schema_version` field:

- `"decision-snapshot-v1"` — current schema (v0.8.0+)
- v0.7.0 snapshots (no version field) are auto-detected and upgraded on load

### Snapshot Lineage

Each decision snapshot is linked via `preceding_snapshot_id`:

```
snapshot_1 (root) → snapshot_2 → snapshot_3 (current)
```

Lineage is stored in `.auteur/decisions/lineage/<decision_id>.json`.

## Consumed Reasoning Data

The `ReasoningAdapter` reads:
- Review files from `chapters/<N>/reasoning/reviews/reasoning_review_*.yaml`
- Run records from `chapters/<N>/reasoning/runs/reasoning_run_*.yaml`
- Latest run pointer from `chapters/<N>/reasoning/latest.yaml`
- Source content hashes for staleness detection

## Authoritative Reconciliation API

The `ReconciliationAdapter` wraps `ReconciliationStore` from
`auteur.expression.reconciliation`. It distinguishes:
- **Technical conflicts** → `NEEDS_RECONCILIATION`
- **Creative author decisions** → `NEEDS_AUTHOR_DECISION`

## Workflow Integration

`WorkflowEngine.analyze()` queries the DecisionWorkspace when a
`decision_service` is provided. Decision actions inject into the
`recommend_actions()` output, ranked by priority:

1. Blocked decisions (highest priority)
2. Author-decision needed
3. Evaluation/comparison
4. Acceptance-ready

When no decisions are open, behavior is identical to v0.7.0.

## Safe vs Author-Required Actions

### May execute automatically

- `auteur decision next <id>` (for generates/evaluates)
- `auteur decision compare <id>`
- `auteur decision conflicts <id>`
- `auteur decision impact-preview <id>`
- `auteur decision inspect <id>`
- `auteur decision status/list/history/lineage/diff`

### Always require author authority

- `auteur decision prepare-acceptance <id> --candidate <id>` (preparation is derived, but blocked from auto-execute)
- Any acceptance, canonical mutation, or creative merge
- Book-level reconciliation (`auteur expression compose-book`, etc.)

## What Remains Deferred

- Automated reconciliation proposal generation
- Proactive impact monitoring (watching for changes without CLI trigger)
- Direct graphical UI (planned for v0.9.0)
