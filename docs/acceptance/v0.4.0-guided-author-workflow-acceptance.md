# v0.4.0 Guided Author Workflow — Acceptance Proof

## Summary

v0.4.0 adds a **Guided Author Workflow** layer that answers "Where am I?
What's next? Is it safe?" over the existing narrative pipeline. It composes
with `auteur.status.gather_status()` and adds typed workflow semantics
without duplicating or replacing existing status logic.

Subprocess safety: structured arg vectors, recursion prevention, shell
metacharacter rejection, placeholder detection (regex-based), state
revalidation before dispatch, and failed-child non-mutation.

## Bug fix: placeholder regex

**Problem:** `_has_placeholders()` used `p.isalpha() and p.isupper()`, which failed
to detect `PATH_TO_REVIEW` because underscores cause `isalpha()` to return False.

**Fix:** Replaced with `re.fullmatch(r"[A-Z_][A-Z0-9_]*", p)`, which correctly matches:
- `PATH_TO_REVIEW` (underscores, all-caps)
- `ACCEPTANCE_ID` (underscores, all-caps)
- `DRAFT_1` (digits, all-caps)
- `<angle_bracket>` bracket (separate check)

**Regression test:** `test_placeholder_uppercase_rejected`, `test_uppercase_placeholder`,
`test_mixed_placeholders` all pass.

## Architecture notes

- `WorkflowState` is typed and serializable via `to_dict()`
- Stage ordering is deterministic (fixed `STAGE_ORDER` list)
- Blocker ordering is deterministic (stage order, then insertion order)
- Action ranking is deterministic (first incomplete stage wins, single action per stage)
- `status.gather_status()` is reused, not duplicated
- Experimental subsystems (Ontology) are not mandatory
- Accepted and candidate artifacts remain distinct
- Reasoning blockers prevent premature acceptance recommendations
- Reconciliation requirements are respected before acceptance actions

## Safe execution boundary

```text
read_only             → eligible (subprocess dispatch)
derived_artifact      → eligible (subprocess dispatch)
candidate_generation  → eligible (subprocess dispatch)
proposal_generation   → NOT eligible in v0.4.0 (reserved, no concrete action uses it yet)
authority_bearing     → refused with actionable error
canonical_mutation    → refused with actionable error
```

**Additional safety layers:**
- Recursion prevention: `workflow` subcommands blocked (`_FORBIDDEN_SUBCOMMAND_PREFIXES`)
- Placeholder rejection: uppercase-with-underscores and `<angle>` args refused
- Shell metacharacter rejection: `;&|`$(){}[]!#~` refused in arguments
- Structured arg vectors: `[auteur_exe] + args`, no `shell=True`, no string interpolation
- State revalidation: `execute()` calls `self.analyze()` before dispatching
- Failed-child non-mutation: failed subprocess does not advance or mutate workflow state
- Command existence validation: `shutil.which("auteur")` before dispatch

## Remaining limitations

- `--execute` requires LLM credentials for derived_artifact/candidate_generation actions;
  produces exit code 2 if API key is not configured (by design — author must configure)
- `proposal_generation` excluded from autonomous execution in v0.4.0; will be re-evaluated
  when a concrete action requires it
- No rollback or undo for `--execute` (author is in full control via `--execute` flag)
- Layer 3 (narrative_realization) `SceneOutline` schema requires `goal` field;
  documented as Partial — 27 pre-existing xfails

The workflow recommends and safely orchestrates existing capabilities.
It does not accept narrative artifacts or mutate canonical authority.
