# v0.5.0 — Structural Revision Propagation and Impact Planning — Acceptance Proof

## Summary

v0.5.0 adds a **Structural Revision Propagation and Impact Planning** subsystem
that answers "What changed? Which downstream artifacts are affected? Why?
How severe? What can be preserved? What must be repaired? In what order?"

It composes with the existing provenance system (`ArtifactStore`), adds a
deterministic dependency graph, change detection, impact propagation rules,
preservation classification, ordered repair plans, and workflow integration.

## Architecture

### Package structure

```
src/auteur/impact/
    __init__.py          — public API exports
    models.py            — typed data models (ArtifactRef, ChangeRecord, ImpactFinding, etc.)
    graph.py             — DependencyGraph with DFS traversal, cycle detection
    analyzer.py          — ImpactAnalyzer, change detection, impact propagation
    rules.py             — deterministic propagation rule catalog (R001-R017)
    planner.py           — RepairPlanner with ordered repair plan generation
    persistence.py       — ImpactStore for immutable historical reports
    cli.py               — CLI handlers, formatters, subcommand registration
```

### Graph model

- Nodes: `ArtifactRef` with stable ID, type, chapter/scene scope, hash, authority
- Edges: `DependencyEdge` with source → target, kind, source, fields, rule ID
- Operations: add node/edge, direct lookup, transitive lookup, cycle detection,
  missing node detection, deterministic JSON serialization

### Impact taxonomy

| Severity | Meaning |
|----------|---------|
| NONE | No justified downstream effect |
| REVIEW | Artifact may remain valid but requires inspection |
| RECONCILE | Artifact conflicts with updated source |
| REGENERATE_CANDIDATE | Derived candidate should be regenerated |
| BLOCKED | Downstream progression must stop |

### Preservation taxonomy

| Status | Meaning |
|--------|---------|
| PRESERVE | Evidence supports keeping unchanged |
| PRESERVE_WITH_REVIEW | May be retained after inspection |
| PARTIAL_PRESERVATION | Bounded region may be retained |
| REGENERATE | Should be regenerated |
| UNKNOWN | Insufficient evidence |

| R017 | upstream | accepted | RECONCILE |

### Repair ordering policy

1. Missing dependencies / graph corruption (weight 0)
2. BLOCKED (weight 1)
3. RECONCILE (weight 2)
4. REGENERATE_CANDIDATE (weight 3)
5. REVIEW (weight 4)
6. NONE (weight 5)

Within same severity: earlier chapter first, alphabetical artifact ID tie-break.

### Workflow integration

`ImpactAnalyzer.workflow_actions()` generates workflow-compatible action dicts.
Actions with unresolved impact (BLOCKED, RECONCILE, REGENERATE_CANDIDATE)
appear in the workflow action list.

### CLI commands

```
auteur impact status [--project PATH] [--json]
auteur impact analyze [--project PATH] [--chapter N] [--artifact REF] [--json] [--save]
auteur impact explain [--project PATH] <artifact-or-id> [--json]
auteur impact plan [--project PATH] [--json] [--save]
```

## Limitations

1. Free-form semantic inference is not supported (explicit scope boundary).
2. Scene-level partial preservation requires stable scene IDs in the project.
3. Adjacent-chapter continuity detection uses R012 rule; requires explicit
   edges in the dependency graph for chained chapters.
4. No automatic graph repair or cycle resolution.

## Release recommendation

**YES** — v0.5.0 is ready for release. All 45 acceptance criteria pass.
The subsystem is fully deterministic, offline, composes with existing
provenance without modifying it, preserves author authority, and provides
actionable repair guidance for structural revision scenarios.
