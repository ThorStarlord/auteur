# v0.6.0 — Realization Convergence and Scene-Level Revision Workflow

## 1. Purpose

Help an author move from:

```
affected structure or realization
→ bounded revision scope
→ one or more repair candidates
→ candidate comparison
→ reconciliation proposal
→ explicit author acceptance
→ refreshed downstream state
```

## 2. Architecture

The convergence subsystem is a **production-grade bounded realization revision**
workflow that composes with existing v0.3–v0.5 infrastructure:

- **provenance** — artifact lifecycle and integrity
- **impact** — structural revision propagation
- **workflow** — guided author workflow
- **realization** — scene/beat realization layer
- **expression** — prose candidate generation
- **reasoning** — evaluation infrastructure
- **reconciliation** — proposal planning and publication
- **acceptance** — canonical pointer movement

### 2.1 Core Invariant

```
candidate generation
≠ reconciliation
≠ acceptance
≠ canonical mutation
```

The system may:
- identify a bounded revision target
- collect source obligations
- preserve unaffected regions
- generate or register candidate realizations
- compare candidates deterministically
- run existing reasoning evaluation
- produce reconciliation proposals
- recommend a next action
- write derived reports and candidate artifacts

The system may NOT:
- overwrite an accepted realization
- silently replace canonical prose
- move accepted pointers
- choose a creative winner automatically
- discard prior candidates
- mutate author-authored sources
- treat a model score as author acceptance
- automatically apply reconciliation proposals

## 3. Key Concepts

### 3.1 Revision Target

A typed identifier for what needs repair:

- **chapter realization** — a full chapter's scene set
- **scene realization** — a single scene
- **scene beat range** — one or more beats within a scene
- **structural obligation** — a declared obligation from structure/identity

Target resolution is deterministic and uses stable IDs (chapter index, scene ID,
beat IDs).

### 3.2 Source Obligation

An explicit requirement the repair must satisfy:

- from story identity (genre, mode, medium contracts)
- from structure (chapter outline, scene purpose, beat declarations)
- from impact findings (what changed and what must be reconciled)
- from character-state inputs
- from setup/payoff references
- from accepted upstream decisions

Obligations are typed as `required` or `advisory`.

### 3.3 Candidate

A derived, noncanonical realization artifact:

- has a stable candidate ID
- has lineage (parent candidate or source artifact)
- has a generation or registration method
- references source obligations and preserved regions
- stores content artifact path
- records provenance
- has timestamps and authority metadata
- tracks canonical status and freshness
- references evaluation results

### 3.4 Lifecycle

```
DRAFT → GENERATED/REGISTERED → EVALUATED → RECONCILIATION_PROPOSED
→ ACCEPTED | REJECTED | SUPERSEDED | STALE
```

### 3.5 Candidate Comparison

Deterministic comparison over structured evidence:

- obligation coverage
- preservation compatibility
- continuity compatibility
- freshness
- reasoning evaluation availability
- reasoning evaluation status
- unresolved blockers
- authority requirements
- candidate lineage

### 3.6 Reconciliation Proposal

A typed proposal that identifies:
- candidate(s) considered
- source obligations
- satisfied and unsatisfied obligations
- conflicts
- continuity risks
- recommended edits or decisions
- authority-required choices
- proposed downstream invalidations

## 5. Integration Points

### 5.1 Impact (v0.5.0)

- ImpactSeverity.RECONCILE → convergence target
- ImpactFinding.affected_artifact → revision target
- RepairAction.preserved_artifacts → preservation input

### 5.2 Workflow (v0.4.0)

- convergence action recommendations → WorkflowAction
- SAFE_AUTHORITIES for safe operations
- AUTHORITY_BEARING for acceptance
- WorkflowBlocker for unresolved convergence

### 5.3 Provenance

- ArtifactStore for sidecar metadata
- Lifecycle for candidate lifecycle (DRAFT → PROPOSED → ACCEPTED)
- canonical_content_hash for content integrity

### 5.4 Expression

- ExpressionStore for prose candidate storage
- ChapterExpressionStore for chapter assembly

### 5.5 Reasoning

- ReasoningRuntime for candidate evaluation
- ArtifactRevision for candidate-specific evaluation

### 5.6 Acceptance

- Existing acceptance APIs for canonical pointer movement
- No parallel acceptance mechanism

## 6. CLI Surface

```
auteur realization status --chapter 3 --scene scene-04
auteur realization revise --chapter 3 --scene scene-04
auteur realization candidates --chapter 3 --scene scene-04
auteur realization generate-candidate --chapter 3 --scene scene-04 --strategy minimal-repair
auteur realization register-candidate --chapter 3 --scene scene-04 --file revised-scene.yaml
auteur realization compare --chapter 3 --scene scene-04
auteur realization reconcile --chapter 3 --scene scene-04
```

All commands support `--project`, `--json`, `--chapter`, `--scene`, `--candidate`,
`--strategy` where applicable.

## 7. Persistence Layout

```
.auteur/convergence/
    targets/
    candidates/
    comparisons/
    proposals/
    latest/
```

Historical records are immutable. Convenience pointers are replaceable atomically.

## 8. Failure Behavior

- Missing project: actionable message, nonzero exit, no traceback by default
- Ambiguous target: list candidates, do not guess
- Missing stable scene boundary: fall back to broader scope, explain
- Candidate generation failure: preserve prior candidates, no partial registration
- Stale candidate: visible status, no silent acceptance
- Reconciliation conflict: preserve proposal, require author decision
- Acceptance failure: no canonical-pointer movement, retain candidate

## 9. Scope Boundaries

v0.6.0 provides production-grade bounded realization revision for:

- chapter realization (stable chapter IDs)
- scene realization (stable scene IDs)
- scene beat range (where stable beat IDs exist)
- explicitly declared structural obligation

Unsupported in v0.6.0:
- arbitrary prose-span repair without stable mappings
- automated creative selection
- hidden automatic merging
