# v0.5.0 — Structural Revision Propagation and Impact Planning

## 1. Mission

Implement a deterministic **Structural Revision Propagation and Impact Planning**
subsystem that answers:

```
What changed?
Which downstream artifacts are affected?
Why are they affected?
How severe is the impact?
What can safely be preserved?
What must be reviewed, reconciled, regenerated, or blocked?
In what order should repair work happen?
What is the next safe workflow action?
```

The system converts existing provenance, freshness, hashes, accepted pointers,
and artifact relationships into an actionable repair plan.

## 2. Architecture

### 2.1 Package structure

```
src/auteur/impact/
    __init__.py          — public API
    models.py            — typed data models
    graph.py             — dependency graph construction
    analyzer.py          — change detection and impact propagation
    rules.py             — deterministic propagation rules
    planner.py           — repair plan generation and ordering
    persistence.py       — impact report persistence
    cli.py               — CLI subcommand registration and rendering
```

### 2.2 Design decisions

1. **Compose with existing provenance**, not replace it. Use `ArtifactStore`
   for content hashing, dependency records, and lifecycle tracking. The impact
   graph is a *derived* view built from existing artifact metadata.
2. **Deterministic throughout**. No LLM calls, no randomization, no
   timestamp-dependent decisions (except recording when analysis was run).
3. **Graph-based propagation**. Build a directed dependency graph of
   artifact-to-artifact edges, then traverse for impact.
4. **Explicit rules**. Impact severity decisions come from a rule catalog, not
   graph distance heuristics.
5. **Preservation is explicit**. Every artifact gets a preservation
   classification. When evidence is insufficient, default to UNKNOWN or REVIEW.
6. **Repair plans are ordered**. Deterministic ordering policy with
   prerequisites, chapter numbering, and workflow stage priority.
7. **Immutable history**. Historical analyses and plans are written once.
   Only a `latest.yaml` convenience pointer is replaced.
8. **Workflow integration**. Impact repair actions appear as workflow
   recommendations when unresolved high-severity impact exists.

## 3. Artifact identity

Every graph node uses a stable artifact identity. The existing
`ArtifactStore._artifact_id()` provides the base identity scheme.

An artifact reference (`ArtifactRef`) distinguishes:

| Field | Description |
|-------|-------------|
| `artifact_id` | Stable identifier (e.g. `story_identity`, `chapter_03`, `scene_03_01`) |
| `artifact_type` | Type (e.g. `story_identity`, `blueprint`, `chapter_outline`, `scene_realization`, `scene_expression`, `chapter_expression`, `book_expression`) |
| `project_scope` | Scope (e.g. `identity`, `structure`, `chapter`, `scene`, `book`) |
| `chapter_index` | Chapter number where applicable |
| `scene_index` | Scene number where applicable |
| `file_path` | Canonical project-relative path |
| `content_hash` | Current content hash |
| `authority` | Authority level (canonical/derived) |
| `accepted` | Whether the artifact has been accepted |

## 4. Dependency graph

The directed graph models artifact-to-artifact dependency edges.

### 4.1 Default dependency chain

```
Identity → Structure → Chapter outline → Scene realization
→ Scene expression → Chapter expression → Book expression
→ Published outputs
```

Plus cross-cutting edges for reasoning reviews, reconciliation results, and
accepted artifacts.

### 4.2 Edge model

Each `DependencyEdge` has:

| Field | Description |
|-------|-------------|
| `source_id` | Upstream artifact ID |
| `target_id` | Downstream artifact ID |
| `kind` | DependencyKind (structural, semantic, state_order) |
| `source` | DependencySource (declared, inferred, generated) |
| `fields` | Affected fields (from provenance projections) |
| `rule_id` | Propagation rule ID that created this edge |

### 4.3 Graph operations

- `add_node(ref)` — register an artifact
- `add_edge(source_id, target_id)` — register a dependency
- `direct_dependencies(id)` → list of direct dependencies
- `direct_dependents(id)` → list of direct dependents
- `transitive_dependents(id)` → all downstream affected nodes
- `has_cycle()` → cycle detection (DFS-based)
- `find_cycle_members()` → set of nodes in cycles
- `missing_nodes()` → unresolved dependency references
- `to_dict()` → stable JSON serialization

## 5. Change detection

Uses existing `canonical_content_hash()` from `auteur.provenance.store`.

### 5.1 Change types

| Type | Detection |
|------|-----------|
| `CONTENT_CHANGED` | Content hash differs from recorded hash |
| `POINTER_CHANGED` | Accepted source pointer differs |
| `ARTIFACT_REMOVED` | File no longer exists |
| `ARTIFACT_ADDED` | New artifact not previously tracked |
| `ACCEPTED_SOURCE_CHANGED` | Accepted artifact's upstream source changed |
| `DEPENDENCY_CHANGED` | Dependency declaration changed |
| `SCHEMA_VERSION_CHANGED` | Schema version metadata changed |

### 5.2 ChangeRecord

| Field | Description |
|-------|-------------|
| `change_id` | UUID-based unique ID |
| `artifact_ref` | The changed artifact |
| `change_type` | Type of change |
| `previous_hash` | Hash before change |
| `current_hash` | Hash after change |
| `detected_at` | ISO timestamp |
| `evidence` | Supporting details |

## 6. Impact taxonomy

### 6.1 Severity values

| Severity | Meaning |
|----------|---------|
| `NONE` | No justified downstream effect |
| `REVIEW` | Artifact may remain valid but requires inspection |
| `RECONCILE` | Artifact conflicts with updated source; needs deliberate comparison |
| `REGENERATE_CANDIDATE` | Derived candidate should be regenerated |
| `BLOCKED` | Downstream progression must stop until resolved |

### 6.2 ImpactFinding

| Field | Description |
|-------|-------------|
| `finding_id` | UUID-based |
| `source_change` | The ChangeRecord causing this |
| `affected_artifact` | The downstream ArtifactRef |
| `is_direct` | True for direct dependency |
| `severity` | ImpactSeverity |
| `rule_id` | Rule that produced this finding |
| `reason` | Human-readable explanation |
| `dependency_path` | List of artifact IDs from source to affected |
| `preservation` | PreservationStatus |
| `recommended_action` | Recommended next step |
| `authority_required` | Authority level needed |

## 7. Preservation taxonomy

| Status | Meaning |
|--------|---------|
| `PRESERVE` | Evidence supports keeping the artifact unchanged |
| `PRESERVE_WITH_REVIEW` | Artifact may be retained after inspection |
| `PARTIAL_PRESERVATION` | A bounded region may be retained (scene/section level) |
| `REGENERATE` | Artifact should be regenerated as a candidate |
| `UNKNOWN` | Insufficient evidence to claim preservation |

## 8. Propagation rules

Rules are deterministic, testable, and stored in `rules.py`.

### 8.1 Core rules

```
R001: Identity changed → Structure: REVIEW or RECONCILE
R002: Structure changed → Chapter outline: RECONCILE
R003: Chapter outline changed → Scene realization: RECONCILE
R004: Scene realization changed → Scene expression: REGENERATE_CANDIDATE
R005: Scene expression changed → Chapter expression: REGENERATE_CANDIDATE
R006: Chapter expression changed → Book expression: REGENERATE_CANDIDATE
R007: Book expression changed → Published outputs: BLOCKED
R008: Any upstream changed → Reasoning review: BLOCKED (stale)
R009: Changed accepted chapter → Book assembly: REGENERATE_CANDIDATE
R010: Changed accepted chapter → Reconciliation result: REVIEW
R011: Changed book assembly → Published outputs: REGENERATE_CANDIDATE
R012: Adjacent chapter changed → Next chapter opening: REVIEW (continuity)
R013: Unrelated chapter → PRESERVE (no dependency path)
R014: Missing dependency → BLOCKED
R015: Cycle detected → BLOCKED for affected component
R016: Accepted artifact (same content) → PRESERVE
R017: Accepted artifact (upstream changed) → RECONCILE
```

### 8.2 Severity propagation

When multiple paths affect the same artifact:
- Preserve all relevant reasons
- Deduplicate equivalent findings
- Choose the highest justified severity
- Retain path evidence
- Keep deterministic ordering

## 9. Repair plan

### 9.1 RepairAction

| Field | Description |
|-------|-------------|
| `action_id` | Stable ID |
| `title` | Short title |
| `description` | Description |
| `affected_artifact` | ArtifactRef |
| `prerequisites` | List of action IDs that must precede this |
| `command` | CLI command template |
| `authority` | AuthorityLevel required |
| `safe_to_execute` | Whether this can be auto-executed |
| `blocking` | Whether this blocks other work |
| `reason` | Why this repair is needed |
| `preserved_artifacts` | What is preserved through this action |

### 9.2 Ordering policy

1. Graph corruption or missing dependencies
2. Direct upstream contradictions (RECONCILE)
3. Reconciliation before regeneration
4. Realization before expression
5. Expression before reasoning
6. Reasoning before acceptance
7. Accepted chapter before book assembly
8. Book assembly before publishing
9. Earliest affected chapter first
10. Stable artifact ID tie-break (alphabetical)

## 10. Workflow integration

When unresolved high-priority impact exists, impact repair actions outrank
unrelated generation or publishing actions.

The `WorkflowEngine.analyze()` method is extended to check for unresolved
impact. If found, impact repair actions are prepended to the action list.

Integration points:
- `ImpactAnalyzer.has_unresolved_impact()` — check if repairs remain
- `ImpactAnalyzer.workflow_actions()` — generate WorkflowAction from repairs
- Repair actions use the same `AuthorityLevel` and safety models

## 11. CLI surface

### 11.1 Commands

```
auteur impact status [--project PATH] [--json]
auteur impact analyze [--project PATH] [--chapter N] [--artifact REF] [--json]
auteur impact explain [--project PATH] <artifact-or-id> [--json]
auteur impact plan [--project PATH] [--json]
```

### 11.2 Output

Default output emphasizes:
- Changed artifacts
- Directly affected artifacts
- Transitively affected artifacts
- Blocked
- Review required
- Can be preserved
- Repair order
- Next safe action
- Authority warning

## 12. Persistence

Impact reports stored under `.auteur/impact/`:

```
.auteur/impact/
    analyses/<analysis_id>.json
    plans/<plan_id>.json
    latest.yaml          — convenience pointer
```

- Historical analyses and plans are immutable (written once)
- Only `latest.yaml` may be replaced
- Reports use project-relative paths
- `authority: derived`, `canonical: false`

## 13. Failure behavior

| Condition | Behavior |
|-----------|----------|
| Missing project | Actionable error, nonzero exit |
| Malformed provenance | Explicit blocker, preserve other results |
| Missing dependency | Report missing node, mark path incomplete |
| Cycle | Identify members, block affected, preserve rest |
| Ambiguous identity | UNKNOWN impact, require inspection |
| Persistence failure | Don't update latest, report error |
| Workflow integration failure | CLI remains usable, workflow reports blocker |

## 14. Scope boundary

**Supported (v0.5.0):**
- Artifact-level propagation for the documented workflow chain
- Explicit semantic references (character arc checkpoints, setup/payoff IDs,
  scene dependency IDs, declared thematic obligations, structure links)

**Deferred/Experimental:**
- Unrestricted free-form semantic inference
- LLM-based dependency detection
- Automatic prose rewriting

## 15. Backward compatibility

- Impact module is additive — no existing code is modified
- Provenance system unchanged
- Workflow system extended (not replaced)
- Existing `auteur workflow next` behavior preserved when no impact exists
