# v0.3 Reasoning Runtime Integration Plan

## Architecture Decision

The ReasoningRuntime will **replace** direct `run_critics()` calls in `PipelineRunner.draft_chapter()`.
The old `run_critics()` function remains importable but is no longer called by production code.

## Design Decisions

### 1. Runtime replaces `run_critics()`
- `PipelineRunner.draft_chapter()` calls `reasoning_runtime.run(request)` instead of `run_critics()`
- The Runtime's sequential execution is acceptable; parallelism can be added later without API change
- Critics execute once through the Runtime, never twice

### 2. ValidationReport compatibility
- After Runtime execution, project outcomes into a `ValidationReport` using the same fields
- `passed` = no error-severity `CriticFinding`s (same logic as today)
- Existing consumers (`write_validation`, `on_iteration` callback, retry loop) receive unchanged `ValidationReport`

### 3. Execution model
- 5 draft critics run sequentially through `ReasoningRuntime.run()` — no dependencies between them
- Synthesis runs after all critics complete
- Deterministic ordering by critic_id alpha sort for reproducibility

### 4. Artifact paths
```
chapters/<NN>/reasoning/
  runs/
    reasoning_run_<chapter>_<iteration>_<id>.yaml
  outcomes/
    <run_id>/
      contract.json
      arc.json
      tension.json
      slop.json
      theme.json
  reviews/
    reasoning_review_<run_id>.yaml
    reasoning_review_<run_id>.md
  latest.yaml
```

### 5. Source snapshot boundaries
- Draft text (content hash)
- Outline (content hash)
- Blueprint revision (content hash)
- Bible revision (content hash)
- Chapter index and iteration

### 6. Failure semantics
- One critic malformed → error finding preserved, overall failed
- One critic exception → FAILED outcome, review shows incomplete
- Persistence failure → no partial visible state (atomic writes via temp file + rename)
- Synthesis failure → raw outcomes preserved, review marked absent

### 7. Atomicity
- Run record written last (after all outcomes + synthesis)
- Temp file + atomic rename for each write
- `latest.yaml` is the only mutable pointer (overwritten each new run)

### 8. Token accounting
- `_CountingClient` wraps LLM as before — Runtime doesn't need to change
- Pipeline's existing `counted_llm.input_tokens/output_tokens` still works

### 9. Retry semantics
- Same retry loop: failed iteration calls `draft_chapter` again with `prior_findings`
- New ReasoningRuntime run created each iteration (independent)
- Prior outcomes remain immutable in `runs/`

### 10. CLI
- `auteur reasoning review <path>` works on persisted reviews
- `auteur reasoning inspect <path> <group>` works on persisted outcomes
- No new top-level reasoning command needed

### 11. Backward compatibility
- `run_critics()` remains importable but is no longer called by PipelineRunner
- `ValidationReport` schema unchanged
- All existing tests continue to pass

### 12. Rollout
1. Create critic adapters (Subagent D)
2. Create persistence/review module (Subagent E)
3. Modify runner.py to use Runtime
4. Test, dogfood, commit
