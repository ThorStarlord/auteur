# v0.7.0 Author Decision Workspace — Design



## 1. Overview



The Author Decision Workspace gives the author one coherent view of what changed, what is affected, what repair targets exist, which candidates are available, what evidence supports each candidate, what conflicts remain, and what requires explicit author authority.



The workspace composes existing capabilities (v0.4 workflow guidance, v0.5 impact analysis, v0.6 convergence candidates, reasoning evidence, reconciliation proposals, acceptance readiness) into a single application-service layer and CLI interface.



**Central invariant:**

```

decision support ≠ creative judgment ≠ acceptance ≠ canonical mutation

```



## 2. Decision Identity



A **decision** is a bounded author-facing task that aggregates evidence and options for resolving one specific narrative problem.



### DecisionID



Decision identity must be:

- **Stable** — same unresolved problem state produces same ID

- **Deterministic** — derived from problem scope, not display text

- **Inspectable** — can be loaded from persistence and examined



Implementation:

```

decision_id = sha256(

  project + "|" +

  trigger_type + "|" +

  scope (chapter_index + scene_id + beat_ids) + "|" +

  target_artifact_id + "|" +

  sorted(impact_finding_ids)

).hex()[:16]

```



This ensures:

- Same narrative problem reproduces same ID

- Different project or scope produces different ID

- Added impact findings may produce different ID

- Display order does not affect ID

- ID is minimal but collision-resistant for typical repository sizes



## 3. Decision Lifecycle



States:

1. **OPEN** — decision exists, evidence incomplete

2. **EVIDENCE_INCOMPLETE** — missing candidate or evaluation

3. **CANDIDATES_REQUIRED** — no viable candidate exists

4. **EVALUATION_REQUIRED** — candidate exists but lacks reasoning review

5. **COMPARISON_REQUIRED** — multiple candidates exist, no comparison

6. **RECONCILIATION_REQUIRED** — comparison exists, conflicts unresolved

7. **AUTHOR_DECISION_REQUIRED** — evidence complete, creative choice needed

8. **READY_FOR_ACCEPTANCE** — all prerequisites satisfied

9. **RESOLVED** — decision accepted

10. **STALE** — source evidence changed after decision created

11. **BLOCKED** — missing dependency, malformed provenance



Lifecycle is **derived**, not stored. A decision is stale if:

- Source impact findings are stale

- Candidate depends on stale structure

- Reasoning evidence is stale

- Impact analysis version changed



## 4. Decision Sources and Triggers



A decision may be triggered by:



1. **ImpactFinding** — a narrative artifact changed, affected others

   - Trigger: `impact_finding_id`

   - Scope: chapter_index, scene_index from finding

   - Target: affected artifact

   

2. **ConvergenceTarget** — explicit revision target exists

   - Trigger: `target_id`

   - Scope: target's scope (chapter/scene/beat)

   - Target: source artifact

   

3. **ReasoningConflict** — evaluation disagreement exists

   - Trigger: `reasoning_conflict_id`

   - Scope: artifact chapter/scene

   - Target: evaluated artifact

   

4. **ReconciliationConflict** — proposal has unresolved conflicts

   - Trigger: `proposal_id`

   - Scope: target's scope

   - Target: reconciliation subject

   

5. **AcceptanceBlocker** — candidate ready but prerequisites unmet

   - Trigger: `blocker_id`

   - Scope: target artifact location

   - Target: candidate



Deduplication: If two triggers would create decisions for the same target and scope, the one with the most complete evidence wins. Historical decisions remain inspectable.



## 5. Decision Model



```python

@dataclass

class AuthorDecision:

    decision_id: str  # stable deterministic hash

    project: str

    chapter_index: int

    scene_id: str | None = None

    beat_ids: list[str] = field(default_factory=list)

    

    # Target and trigger

    target_artifact_id: str

    trigger_type: DecisionTrigger  # enum

    trigger_ids: list[str]  # source finding/target/conflict IDs

    

    # Evidence

    candidates: list[CandidateSummary]

    evidence: list[DecisionEvidence]

    conflicts: list[DecisionConflict]

    unresolved_choices: list[UnresolvedChoice]

    blockers: list[str]

    

    # State

    readiness: DecisionReadiness

    lifecycle_state: LifecycleState

    freshness: EvidenceFreshness

    

    # Authority

    authority_required: AuthorityLevel

    safe_actions: list[str]

    

    # Metadata

    created_at: datetime

    last_updated_at: datetime

    source_snapshot: dict[str, Any]  # capture source state

```



## 6. Evidence Model



Evidence is normalized from all subsystems with explicit provenance.



```python

@dataclass

class DecisionEvidence:

    evidence_id: str  # uuid

    source_subsystem: EvidenceSource  # impact, convergence, reasoning, reconciliation

    source_artifact_id: str  # what created this evidence

    

    claim: str  # the assertion

    evidence_type: EvidenceType  # FACT, STRUCTURAL_FACT, IMPACT_FINDING, etc.

    classification: EvidenceClassification  # FACT, DERIVED_INFERENCE, RECOMMENDATION, AUTHOR_CHOICE

    

    freshness: EvidenceFreshness  # current, stale, unknown

    confidence: str | None  # from source if available

    supporting_reference: str | None  # link to full evidence

    

    candidate_id: str | None  # if evidence is candidate-specific

    authority: AuthorityLevel

    

    created_at: datetime

```



Classification examples:

- **FACT**: "Candidate A depends on structure hash abc123"

- **DERIVED_INFERENCE**: "Candidate A is stale because current structure hash differs"

- **RECOMMENDATION**: "Regenerate or reconcile Candidate A before acceptance"

- **AUTHOR_CHOICE**: "Decide whether to preserve the original ending"



## 7. Candidate Summaries



For each candidate, provide multidimensional view without copying full content:



```python

@dataclass

class CandidateSummary:

    candidate_id: str

    status: CandidateStatus

    freshness: EvidenceFreshness

    lineage: CandidateLineage  # parent/generation method

    

    obligations_satisfied: list[str]

    obligations_unsatisfied: list[str]

    preserved_regions: list[str]

    reasoning_evidence: list[str]  # reference IDs

```



No single "score" or "best candidate" field. Candidates are multidimensional.



## 8. Unresolved Choices



Author decisions are explicit:



```python

@dataclass

class UnresolvedChoice:

    choice_id: str  # uuid

    question: str  # the decision point

    options: list[ChoiceOption] | None  # None = open-ended

    

    affected_candidates: list[str]  # which candidates this changes

    supporting_evidence: list[str]  # evidence references

    tradeoffs: list[str]  # what is gained/lost per option

    

    required_authority: AuthorityLevel

    blocking_status: bool  # blocks acceptance if true

    

    created_at: datetime

```



Examples:

- "Preserve the accepted opening or replace the full scene?"

- "Allow Marcus to infer Elena's suspicion?"

- "Choose candidate A, candidate B, or request another revision?"

- "Resolve conflicting required obligations?"



## 9. Readiness Classification



Readiness is deterministic and based on what is missing or stale.



```python

class DecisionReadiness(str, Enum):

    BLOCKED = "blocked"  # malformed or missing critical data

    NEEDS_CANDIDATE = "needs_candidate"

    NEEDS_EVALUATION = "needs_evaluation"  # candidate lacks reasoning

    NEEDS_COMPARISON = "needs_comparison"  # multiple candidates, no comparison

    NEEDS_RECONCILIATION = "needs_reconciliation"  # conflicts detected

    NEEDS_AUTHOR_DECISION = "needs_author_decision"  # all evidence present, choice required

    READY_FOR_ACCEPTANCE = "ready_for_acceptance"  # all prerequisites met

    RESOLVED = "resolved"  # accepted

    STALE = "stale"  # source changed

```



Rules (in order):

1. If critical data missing/malformed → **BLOCKED**

2. If target ambiguous → **BLOCKED**

3. If evidence is stale → **STALE**

4. If no candidate exists → **NEEDS_CANDIDATE**

5. If candidate exists but stale → **NEEDS_CANDIDATE**

6. If candidate lacks reasoning evaluation → **NEEDS_EVALUATION**

7. If multiple candidates, no comparison → **NEEDS_COMPARISON**

8. If comparison exists but conflicts unresolved → **NEEDS_RECONCILIATION**

9. If unresolved author choice exists → **NEEDS_AUTHOR_DECISION**

10. If all prerequisites met → **READY_FOR_ACCEPTANCE**



A decision is NOT ready merely because one candidate exists.



## 10. Deterministic Next Actions



Recommend one primary action plus ordered action list.



```python

@dataclass

class DecisionAction:

    action_id: str

    title: str

    reason: str

    command: str | None  # CLI command to execute

    prerequisites: list[str]  # must complete first

    safe_to_execute: bool

    authority_level: AuthorityLevel

    expected_result_state: str  # what readiness will be after

```



Action ordering preference:

1. Fix malformed provenance

2. Resolve ambiguous target

3. Refresh stale impact

4. Generate missing candidate

5. Regenerate stale candidate

6. Evaluate candidate

7. Compare candidates

8. Resolve reconciliation conflict

9. Make author choice

10. Accept ready candidate

11. Refresh downstream



## 11. Acceptance Preparation



Verify readiness without accepting:



```python

@dataclass

class AcceptancePreparation:

    decision_id: str

    candidate_id: str

    is_ready: bool  # bool, not enum

    blockers: list[str]  # reasons for not-ready

    verification_results: dict[str, bool]

    

    # Facts about this acceptance

    will_change_canonical: bool  # false for all convergence candidates

    affected_downstream: list[str]  # impact findings expected

    

    prepared_at: datetime

```



Operation:

- Verify candidate exists and is not stale

- Verify all required obligations satisfied

- Verify required evaluations complete

- Verify reconciliation conflicts resolved

- Verify author choices made

- Return blockers if any fail



Does not accept. Does not move pointer. Does not mutate accepted prose.



## 12. Workflow Integration



When open decisions exist, surface the highest-priority decision action.



Integration level: **Action recommendation**, not full state-machine control.



Decision action maps to WorkflowAction:

```

NEEDS_CANDIDATE → "Generate candidate for [target]"

NEEDS_EVALUATION → "Review candidate evaluation"

NEEDS_COMPARISON → "Compare candidate options"

NEEDS_RECONCILIATION → "Resolve reconciliation conflict"

NEEDS_AUTHOR_DECISION → "Make author decision: [question]"

READY_FOR_ACCEPTANCE → "Prepare acceptance for [candidate]"

```



Requirements:

- Deterministic ranking (readiness state)

- No recursion

- State revalidation before execution

- Safe actions may be auto-executed

- Authority-bearing actions refused

- Failed child command does not advance state

- No decisions preserves existing workflow



## 13. Safe Execution Boundary



Safe unattended actions:

- Decision inspection

- Evidence aggregation

- Candidate listing

- Impact refresh (read-only)

- Reasoning evaluation (generate, no approval)

- Comparison generation

- Reconciliation proposal generation

- Acceptance-readiness check



Unsafe actions (require authority):

- Candidate acceptance

- Canonical pointer movement

- Accepted prose replacement

- Historical candidate deletion

- Creative merge application



Reuse existing AuthorityLevel and safe-execution safeguards from v0.4.



## 14. Persistence



Store immutable decision snapshots. Suggested location:

```

.auteur/

  decisions/

    snapshots/

      decision-id-1.json (immutable source snapshot)

      decision-id-2.json

    latest/

      latest.json (points to current highest-priority decision)

      latest-by-target/

        target-artifact-id.json (latest decision for this target)

```



Rules:

- Snapshots are immutable

- Latest pointers update atomically (temp-file + rename)

- Source artifacts never copied as authority-bearing data

- Historical decisions remain inspectable

- Stale decisions remain visible

- Identical decision state may be idempotent

- Same ID with conflicting content fails explicitly



## 15. CLI Surface



Commands (using `auteur decision` prefix):



```bash

auteur decision status                          # List all open decisions

auteur decision list [--chapter N] [--json]     # List with filters

auteur decision inspect <decision-id>           # Full decision detail

auteur decision evidence <decision-id>          # Evidence breakdown

auteur decision compare <decision-id>           # Candidate comparison

auteur decision next <decision-id>              # Recommended action

auteur decision prepare-acceptance <decision-id> \

  --candidate <candidate-id>                    # Check acceptance readiness

auteur decision workflow-next                   # Highest-priority action for workflow

```



Common flags:

- `--project` — explicitly specify project

- `--chapter` — filter by chapter

- `--scene` — filter by scene

- `--candidate` — candidate ID for operations

- `--json` — structured output

- `--save` — persist output



## 16. Human Output Format



Default emphasizes:

- Decision (title, trigger)

- Target (what needs fixing)

- Readiness (what's missing)

- Candidates (options, status)

- Evidence (facts, conflicts)

- Unresolved choices (creative decisions needed)

- Blockers (what prevents progress)

- Recommended next action

- Authority boundary



Example:

```

Decision: Repair Chapter 3 / Scene 4

Trigger: Chapter outline changed after realization acceptance

Target: scene_03_04_realization



Readiness: NEEDS_AUTHOR_DECISION



Candidates:

  candidate-a — fresh, evaluated, 5/6 obligations satisfied

  candidate-b — fresh, evaluated, all obligations satisfied

  candidate-c — stale after structure update



Evidence:

  candidate-a preserves beats B01–B03

  candidate-b satisfies all required obligations

  candidate-b changes Marcus's knowledge state

  candidate-c depends on stale structure



Conflict:

  candidate-b changes Marcus's knowledge state (new in this revision)



Author choice:

  Preserve secrecy or strengthen dramatic escalation?



Next:

  Review reconciliation proposal for candidate-b



Authority:

  No canonical or accepted content will change automatically.

```

## 18. Failure Behavior



- **Missing project**: actionable error, nonzero exit, no traceback

- **No open decisions**: successful empty state, explain normal workflow continues

- **Ambiguous target**: list candidates, do not guess

- **Missing source subsystem**: preserve available evidence, report incomplete decision

- **Stale evidence**: label stale, block readiness where required

- **Conflicting evidence**: preserve both claims, expose conflict

- **Persistence failure**: preserve previous latest pointer, don't claim durable success

- **Acceptance preparation failure**: no canonical mutation, return blockers



## 19. Compatibility



- v0.4 workflow unchanged if no decisions exist

- v0.5 impact analysis reused as-is

- v0.6 convergence candidates unchanged

- No subsystem duplication

- Backward compatible with existing projects



## 20. Deferred Scope



- LLM-backed decision recommendation (ranking candidates by artistic merit)

- Automatic freshness recomputation on every subsystem update

- GUI interface (architecture allows this without redesign)

- Beat-range CLI shorthand (scene-level primary)

- Author conversation history persistence
