# ADR 002: Shared StructureProposal Format Across Audit and Structure Paths

## Status

Accepted

## Context

Auteur has two diagnostic paths that both produce Decision Packets:

1. **Structure path** — `auteur structure propose-repairs` runs `analyze_structure` against a `StoryBlueprint` and emits proposals that, when resolved, are merged into the blueprint via `auteur structure apply`.

2. **Bible audit path** — `auteur audit --repair` runs `audit_bible_locations` (and future rules) against a `StoryBible` and emits proposals that, when resolved, stamp the YAML in-place. No blueprint mutation occurs.

Both paths write YAML files to `structure/proposals/`. Before this ADR, the `StructureProposal` schema had no field indicating which path produced the artifact, and the two resolution commands were not self-discoverable from the artifact alone.

### The problem

An author (or agent) reading a `structure/proposals/repair_*.yaml` file cannot determine from the artifact which command resolves it:

- `auteur structure apply` merges `option.data` into a new blueprint file.
- `auteur audit --accept` stamps `selection`/`decision` in the YAML only — no blueprint mutation.

Calling the wrong command either silently succeeds (no-op) or produces an incorrect blueprint.

## Decision

We use a single `StructureProposal` YAML format for both paths, with a new optional field:

```yaml
source_domain: "structure"   # or "bible_audit", or null for pre-ADR artifacts
```

- `source_domain: "structure"` — generated by `auteur structure propose-repairs`. Resolved via `auteur structure apply`.
- `source_domain: "bible_audit"` — generated by `auteur audit --repair`. Resolved via `auteur audit --accept`.
- `source_domain: null` — grandfathered artifact from before this ADR. Resolution path must be inferred from filename prefix or context.

Both generation functions populate `source_domain` at creation time. The field is optional with `None` as the default so that existing YAML artifacts validate without migration.

### Alternative considered

**Separate YAML schemas per domain** — a `StructureProposal` schema for the structure path and a `BibleAuditProposal` schema for the audit path, stored in separate directories (e.g., `structure/proposals/` vs `audit/proposals/`).

Rejected because:
- It doubles the proposal-tooling surface (two parsers, two apply functions, two CLI flags).
- The artifact format is structurally identical; only the resolution act differs.
- A single format with a discriminant field is cheaper to maintain and easier to explain in the CONTEXT.md glossary.

## Consequences

- **Self-describing artifacts.** An author or agent can determine the correct resolution command from the artifact alone.
- **Backward compatibility.** Artifacts without `source_domain` parse as `None` — no migration needed.
- **Weak enforcement.** The field is informational; the CLI does not yet reject mismatched `source_domain` values. Enforcement is a follow-on slice.
- **Two resolution semantics remain.** This ADR documents the difference; it does not eliminate it. A future ADR may decide whether to unify them.
